# 草莓订单管理系统协作规则

## 项目定位

- 项目路径固定为：`/Users/gd/Desktop/草莓订单管理系统`
- 项目开发日志固定写入：`/Users/gd/Library/Mobile Documents/iCloud~md~obsidian/Documents/项目管理/草莓订单管理系统--副业/开发/`
- 这是持续迭代中的桌面订单管理系统，默认保留现有业务逻辑和已录入数据，不做破坏性修改。

## 沟通与输出

- 全程中文。
- 面向业务场景优先用大白话解释，再补技术细节。
- 用户如果提到“审查”“同步一下”“收尾”，要先检查项目根 `AGENTS.md`、`README.md`、`docs/` 和当天开发日志是否仍与代码一致。

## 开发红线

- 不要在旧路径 `/Users/gd/Desktop/d` 上继续操作；统一只在当前项目路径工作。
- 不要删除或重置用户已有历史数据，尤其是 `history.json`、配置文件、自动拍单状态、财务记录。
- 任何改动先看当前代码状态，避免重复改已经确认过的东西。
- 如果只是本地环境、构建产物、缓存目录造成 Git 噪音，优先通过 `.gitignore` 收口，不要误当业务改动处理。

## 业务口径

- 售后规则：
  - `仅退款 / 部分退款`：冲减收入
  - `退货退款`：默认按全额退款处理
  - `是否收回货品=是`：采购成本归零
  - `是否收回货品=否`：采购成本保留
- 财务报表口径：
  - 大盘拆分为`本月经营 / 今日经营`
  - 每日账目明细和`今日经营`保持一致
  - 本月视图固定显示当天，即使没数据也显示 `0`
- 历史订单详情区默认结构：
  - 顶部操作栏
  - 售后处理
  - 财务信息
  - 采购信息

## 录单与历史页规则

- 录单默认值按平台分开处理：
  - `录单默认平台`
  - `抖店默认店铺`
  - `微信默认店铺`
- 历史工作台左侧订单列表默认按 `下单时间` 倒序；缺失或非法时可回退 `created_at`，但不要伪造真实下单时间。
- `仅存历史` 允许保存不完整订单；`确认写入飞书` 前必须校验下单时间格式。
- 下单时间允许：
  - `YYYY-MM-DD HH:MM`
  - `YYYY-MM-DD HH:MM:SS`

## 多订单识别规则

- 多订单截图优先先切单，再逐单识别。
- 单个订单失败不能拖垮整张图；UI 要明确提示成功单数和失败单数。
- SKU 图优先使用当前订单自己的切图，不要串到别的订单。
- 收件人识别要兼容中文名、英文名、拼音名和带虚拟号尾号的形式。

## 自动拍单规则

- 自动拍单模块是执行主场；录单页和历史页只做轻入口和轻反馈。
- 第一版结果优先回写本地历史，不修改飞书表结构。
- 如果桥接或真实执行器未就绪，优先给清晰中文提示，不伪造成功。

## 桌面 UI 规则

- macOS 桌面应用固定使用浅色模式，不跟随系统深色模式切换。
- 如果后续新增 Qt 容器控件、tab pane、scroll viewport、splitter 或图表组件，要显式补齐浅色背景，避免出现黑色横条、黑色竖条、黑色 tooltip 或漏底问题。
- UI 外观修复优先从应用启动层 palette、全局 QSS 容器层、组件私有背景三层一起排查，不要只改某个页面的局部卡片背景。

## 文档维护

- `README.md` 负责新人快速启动、测试、功能入口说明。
- `docs/` 负责结构化说明，例如手机录单接入、订单数据结构、UI 预览。
- 开发日志按天维护，同一天多轮开发追加到同一个日期文件。
- 文档里涉及时间时优先用绝对日期；不要把“今天/昨天/最近”这类相对时间写成项目事实。

# 草莓订单管理系统设计稿

## 1. 目标

做一个面向日常拍单和订单录入的桌面工具，服务两个并列工作流：

1. 拍单识别工作流：用户粘贴截图、拖拽图片或选择图片文件后，系统自动识别订单信息，先生成订单卡供人工确认，再写入飞书。
2. 地址快速提取工作流：用户输入地址原文后，系统快速生成两段结果，便于拍单和备注复制。

第一版优先目标不是覆盖所有平台和所有订单模板，而是把当前这类订单截图和地址格式做稳，形成可长期扩展的骨架。

## 2. 设计原则

- 首页必须同时保留两个核心模块：`拍单识别` 和 `地址快速提取`。
- `粘贴截图` 是拍单识别的主入口，同时支持拖拽图片和选择图片文件。
- 系统自动完成识别链路，但不会直接写飞书，必须先展示订单卡给用户确认。
- 视觉风格采用用户确认过的方向：白底、高留白、蓝色重点、品牌红色强调。
- 品牌呈现固定为：
  - 主品牌：`草莓`
  - 副标题：`订单管理系统`

## 3. 页面结构

### 3.1 订单录入页

首页为双模块并排布局：

#### 模块 A：拍单识别

- 输入方式：
  - 粘贴截图
  - 拖拽图片
  - 选择图片文件
- 处理方式：
  - 收到图片后自动开始识别
  - 执行 OCR
  - 提取订单信息
  - 提取或生成地址相关信息
  - 合并为结构化订单卡
- 输出方式：
  - 先展示订单卡
  - 用户确认后写入飞书
  - 写入失败时保留历史记录

#### 模块 B：地址快速提取

- 输入一段原始地址文字，例如：
  - `何女士[3612]15781304332四川省成都市金牛区营门口街道友谊花园9-2304[3612]`
- 输出两段结果：
  - 结果 1：`何女士15781304332四川省成都市金牛区营门口街道友谊花园9-2304`
  - 结果 2：`请电话送货上门谢谢【3612】`
- 页面中提供单独复制按钮。

### 3.2 历史页

历史页用于查看：

- 已写入订单
- 待重试订单
- 处理中订单

后续支持：

- 按日期筛选
- 按状态筛选
- 按手机号或编号搜索
- 查看单条订单的识别结果和写入日志
- 一键重试写入飞书

### 3.3 设置页

设置页统一管理外部连接和字段配置：

- OCR API
- 辅助提取 API
- 飞书 App ID / App Secret
- 目标飞书表格
- 字段映射模板

## 4. 拍单识别流程

### 4.1 输入阶段

用户执行任一动作：

- 粘贴截图
- 拖拽图片
- 选择图片文件

如果输入的是截图图片，则系统自动开始识别任务，不要求用户再点额外按钮。

### 4.2 自动识别阶段

第一版识别链路按以下顺序组织：

1. 对图片执行 OCR。
2. 从 OCR 文本中提取订单基础信息。
3. 从 OCR 文本中提取收货信息。
4. 生成自动备注。
5. 合并成订单卡。

后续如果用户提供更强的图片识别 API 或辅助提取 API，则替换第 1 步或第 2-4 步的实现，不改变整体交互流程。

### 4.3 订单卡确认阶段

系统先展示订单卡，而不是直接写飞书。

订单卡至少展示以下字段：

- 收件人
- 手机号
- 编号
- 收货地址
- 订单编号
- 下单时间
- 商品名称
- 数量
- 订单金额
- 商家收入金额
- 订单状态
- 自动备注

用户确认无误后，点击一次按钮再写入飞书。

## 5. 地址快速提取规则

### 5.1 输入格式

当前确认支持的文本模式：

- 姓名 + `[编号]` + 手机号 + 地址 + `[编号]`

例如：

- `何女士[3612]15781304332四川省成都市金牛区营门口街道友谊花园9-2304[3612]`

### 5.2 输出规则

给定上面的输入，系统固定生成：

- 输出 1：移除前后编号后的 `姓名 + 手机号 + 地址`
- 输出 2：`请电话送货上门谢谢【编号】`

### 5.3 当前已确认示例

输入：

- `何女士[3612]15781304332四川省成都市金牛区营门口街道友谊花园9-2304[3612]`

输出：

- `何女士15781304332四川省成都市金牛区营门口街道友谊花园9-2304`
- `请电话送货上门谢谢【3612】`

## 6. 飞书入表规则

### 6.1 入表策略

- 识别完成后不自动提交。
- 只有在用户确认订单卡后，才写入飞书。
- 如果写入失败，则记录本地历史，允许后续重试。

### 6.2 已确认字段映射方向

针对当前示例订单图，第一版映射方式为：

- 备注：自动备注
- 订单日期：下单日期
- 下单时间：下单时间
- 订单状态：订单状态
- 收入：`商家收入金额`
- 发货地址：姓名 + 手机号 + 编号 + 地址的组合文本
- 价格：订单金额

当前缺失信息允许留空：

- 处理人
- 主店员1
- 产品链接
- 快递单号

### 6.3 已确认收入规则

飞书表中的 `收入` 字段，默认写入 `商家收入金额`，不是订单金额。

## 8. 容错与提示

- 如果截图里商品名被截断，订单卡中仍展示当前识别值，但提示该字段可能不完整。
- 如果缺少关键字段，例如手机号、编号或订单编号，系统不直接提交飞书，而是提示用户补充或修改。
- 如果 OCR 与辅助提取结果冲突，优先以最终结构化规则输出为准，并在订单卡中让用户确认。

## 9. 第一版范围

第一版明确包含：

- 桌面应用
- 首页双模块：
  - 拍单识别
  - 地址快速提取
- 历史页
- 设置页
- 单条订单识别并写入一张固定飞书表
- 本地历史记录

第一版明确不包含：

- 批量订单同时处理
- 多平台订单模板自动切换
- 多飞书表动态路由
- 无确认直接自动写飞书

## 10. 测试思路

第一版开发时，至少验证以下场景：

1. 地址快速提取示例可稳定输出两段结果。
2. 粘贴截图可成功触发识别任务。
3. 订单截图可生成订单卡。
4. 订单卡确认后可写入飞书。
5. 飞书写入失败时可保留历史并支持重试。
6. 缺字段时不可直接提交，必须提示用户确认或修改。

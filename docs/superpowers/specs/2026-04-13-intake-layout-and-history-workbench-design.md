# 草莓订单管理系统：订单录入布局与历史工作台升级设计

## 目标

这一轮升级解决两个问题：

1. `订单录入` 页面信息堆叠过长，字段、采购、地址提取和操作按钮挤在同一纵向流里，录单时不够清晰。
2. `历史` 页面过于简陋，只有简单列表，无法承载“仅存历史”和“确认写入飞书”的完整回看、编辑、删除和再次同步需求。

目标是把这两块升级成真正可长期使用的工作台，同时保持现有白蓝风格和操作习惯不被打断。

## 设计范围

本次只覆盖以下内容：

- `订单录入` 页的结构重排与交互增强
- `历史` 页的结构升级、详细记录展示、编辑、删除、重新写入飞书
- 历史数据结构补充，确保能记录同步来源、同步结果和订单详情快照

本次不做：

- 新的 OCR 能力改造
- 新的飞书权限能力扩展
- 多用户协作或登录系统

## 方案概览

### 订单录入页

页面保持双栏，但重新组织为更清晰的三段：

- 顶部 `识别工具条`
  - 店铺选择
  - `粘贴截图`
  - `选择图片`
  - 当前识别状态
- 左侧 `订单卡`
  - 基础订单信息采用双列表单，减少纵向长度
  - 商品名称、收货地址、自动备注使用强调块，便于快速扫读
  - 采购信息单独成块，固定 3 行
  - 底部保留 `仅存历史` 与 `确认写入飞书`
- 右侧 `地址快速提取`
  - 保持输入、提取、一键复制两段结果
  - 与左侧视觉层级对齐

### 历史页

历史页升级为 `左列表 + 右详情` 工作台：

- 左侧 `记录列表`
  - 展示店铺、收件人、同步方式、状态、时间、订单号
  - 列表项支持选中当前记录
- 右侧 `详情面板`
  - 订单基础信息
  - 采购信息
  - 地址提取结果快照
  - 同步轨迹和错误信息
  - 操作按钮：`编辑记录`、`删除记录`、`重新写入飞书`

## 历史记录模型

历史记录不再只保存摘要，而是保存一份“当时提交/保存时的快照”。每条记录新增或明确保存：

- `record_id`
- `shop_name`
- `sync_source`
  - `仅存历史`
  - `确认写入飞书`
- `status`
  - `仅存历史`
  - `已写入飞书`
  - `写入失败`
- `order_snapshot`
  - 订单编号
  - 下单时间
  - 订单状态
  - 商品名称
  - 数量
  - 商家收入
  - 收件人
  - 手机号
  - 编号
  - 收货地址
  - 自动备注
  - 采购 1-3
- `address_snapshot`
  - 提取结果一
  - 提取结果二
- `created_at`
- `last_synced_at`
- `feishu_result`
  - 成功时可记录 record id
  - 失败时可记录错误文本

旧历史记录继续兼容读取。若旧记录缺字段，详情页按 `-` 展示。

## 编辑与删除

### 编辑

历史页允许编辑当前选中记录的可变字段，至少包括：

- 商品名称
- 商家收入
- 收件人
- 手机号
- 编号
- 收货地址
- 自动备注
- 采购 1-3 的商品、数量、成本

编辑后支持：

- `保存修改`
- `重新写入飞书`

历史页中的编辑不反向修改 OCR 原始结果，只更新历史快照。

### 删除

删除为本地历史删除：

- 删除后从历史文件移除该记录
- 删除不影响飞书中已存在的数据
- 若当前记录已写入飞书，删除前应有明确确认

## 重新写入飞书

历史页中对记录执行 `重新写入飞书` 时：

- 使用该条记录当前保存的订单快照
- 使用当前店铺配置和字段映射
- 成功后更新：
  - `status = 已写入飞书`
  - `last_synced_at`
  - `feishu_result`
- 失败后更新：
  - `status = 写入失败`
  - `feishu_result = 错误信息`

这一步不新建历史记录，而是更新原记录，避免一条订单产生重复历史。

## 数据流

### 订单录入

1. 用户截图识别或手动调整订单
2. 用户可补充采购信息
3. 用户点击：
   - `仅存历史`：写入一条本地历史快照
   - `确认写入飞书`：先写飞书，再将结果写回同一条历史记录

### 历史工作台

1. 页面加载历史记录列表
2. 选中一条记录
3. 右侧展示该记录完整详情
4. 用户可编辑、删除、重新写入飞书

## UI 细节

### 订单录入页

- 基础字段改为双列网格，减少空白浪费
- 商品名称、收货地址、自动备注使用高亮卡片
- 采购信息使用单独区块标题，与订单信息视觉断层清晰
- 地址提取结果卡片保留复制按钮，但整体宽度与左栏更平衡

### 历史页

- 列表项使用状态胶囊区分 `仅存历史 / 已写入飞书 / 写入失败`
- 详情面板中将信息按 `订单信息 / 采购信息 / 同步轨迹` 分组
- 编辑状态下使用与录单页一致的输入组件，避免学习成本

## 错误处理

- 若历史记录缺少店铺配置，`重新写入飞书` 直接提示原因
- 若店铺仍存在但字段映射缺失，沿用当前飞书错误提示逻辑
- 删除失败或历史文件损坏时，页面应显示清晰提示，不静默失败

## 测试策略

至少覆盖：

- 订单录入页新布局仍能正常提交与仅存历史
- 历史记录保存完整快照
- 历史页能渲染详细记录
- 历史页可编辑并保存
- 历史页可删除记录
- 历史页可对现有记录重新写入飞书
- 旧历史文件兼容读取

## 实施拆分

建议按三步落地：

1. 重构 `订单录入` 布局，不改主流程行为
2. 升级 `历史` 存储结构和详情展示
3. 增加 `编辑 / 删除 / 重新写入飞书`

这样每一步都可单独验证，不会把识图、飞书和历史同时搅在一起。
